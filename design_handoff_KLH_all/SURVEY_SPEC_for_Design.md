# สเปกการทำงาน: KLH Data Survey (สำหรับคุยกับ Designer)
> โมดูล "สำรวจ/อัปเดตข้อมูลสินค้า" — ใช้งานจริงบน **มือถือ** (เดินสำรวจหน้างาน/หน้าชั้น)
> เป้าหมายเอกสาร: บอก Designer ว่าหน้านี้ "ทำอะไรได้บ้าง" เพื่อออกแบบ UI มือถือครอบของเดิม — **ห้ามแก้ logic เดิม**
> อ้างอิงโค้ดจริง: `Index.html` (route `?page=survey`)

---

## 1. บริบทการใช้งาน (สำคัญต่อดีไซน์)
- **อุปกรณ์หลัก = มือถือ** ถือเดินสำรวจ ถ่ายรูปสินค้า สแกนบาร์โค้ด เคาะราคา
- มือเดียว/นิ้วโป้ง, กลางแจ้ง (แสงจ้า → ต้องคอนทราสต์สูง), ปุ่มต้องใหญ่กดง่าย
- ทำงานบน Google Apps Script WebView (มี SweetAlert popup, กล้องผ่าน `<input capture>`)

---

## 2. โครงการทำงาน 3 ขั้น (Flow)
```
[ค้นหา/รายการสินค้า]  →  [ฟอร์มแก้ไขสินค้า 1 ตัว]  →  [บันทึก]
   (mock มีแล้ว)            (mock ยังไม่ได้วาด ★)          (popup สำเร็จ)
```
> ★ **ดีไซน์ mock ปัจจุบันมีแค่หน้า "รายการ" — ส่วนที่เป็นหัวใจคือ "ฟอร์มแก้ไข" ยังไม่มีในดีไซน์ ต้องออกแบบเพิ่ม**

## 3. หน้าที่ 1 — รายการ/ค้นหา (ตรงกับ mock)
องค์ประกอบ:
- ปุ่มหัวจอ: **สแกนสำรวจ** (เปิดกล้องสแกนบาร์โค้ดเพื่อเด้งไปสินค้านั้น), **ดึงจาก KLH DATA**
- **KPI 4 ใบ:** สินค้าทั้งหมด / ข้อมูลครบ / รอสำรวจ-อัปเดต / % มีบาร์โค้ด
- ช่องค้นหา (ชื่อ/รหัส/บาร์โค้ด) + ฟิลเตอร์ "เฉพาะรอสำรวจ"
- ตาราง: รหัส · ชื่อ · ทุน · ปลีก · บาร์โค้ด✓ · รูป✓ · สถานะ(ครบ/รออัปเดต)
- **แตะแถว → เปิดฟอร์มแก้ไข (หน้าที่ 2)**

⚠️ ต้องนิยามให้ Designer:
- "สถานะครบ" = มีอะไรครบบ้าง (ปัจจุบันเงื่อนไขจริงยังไม่เข้ม — แค่ "มีบาร์โค้ดชิ้น" ก็ Save ได้)
- KPI ตัวเลขต้องมาจาก backend (ปัจจุบันยังไม่มีฟังก์ชันนับ — เป็นค่า mock)

| ปลีกคำนวณ (X) | X | readonly + ▲▼ | คำนวณ (ดู 6.2) |

### บล็อก C — รูปภาพ & บาร์โค้ด
- ปุ่ม **ถ่ายรูป** (กล้อง) → ครอป 1024×1024 พื้นขาว → preview
- ปุ่ม **สแกนบาร์โค้ด** (กล้อง)
- บาร์โค้ดชิ้น (A) — **บังคับกรอก ถึงจะกด Save ได้**
- บาร์โค้ดลัง (AB) — ถ้าเว้นว่าง auto = บาร์โค้ด+"-B"
- วันที่อัปเดต (Y)
- ปุ่ม **🚀 บันทึกเข้าระบบ** (disabled จนกว่าจะมีบาร์โค้ดชิ้น)

## 5. Interaction พิเศษ (ต้องคงพฤติกรรม)
1. **ปุ่ม ▲▼ ปรับราคา ทีละ 0.25 แบบ snap** — ถ้าราคาไม่ลงตัว .25 จะปัดเข้าหา 0.25 ที่ใกล้สุดก่อน (เช่น 29.11 กด▲→29.25, กด▼→29.00) แล้วครั้งถัดไป ±0.25 ปกติ
2. **☑ โหล** — ติ๊กแล้วคำนวณราคาโหล + auto บาร์โค้ดโหล
3. **ปุ่ม ⚙️ ข้าง dropdown** — เพิ่ม/แก้รายการ master ได้หน้างาน
4. **กล้อง** — ถ่ายรูป (capture) + สแกนบาร์โค้ด
5. แก้ตัวเลขในป๊อปคำนวณต้นทุน → ราคา V/X/โหล อัปเดตตามทันที

## 6. สูตรคำนวณ (logic เดิม — ห้ามเปลี่ยน ค่า)
### 6.1 ต้นทุนสุทธิ/ชิ้น (R) — จาก popup คำนวณ
```
I = ราคาซื้อ/ชิ้น (ถ้ากรอกราคาหน่วยใหญ่ → ÷ ตัวคูณ)
K = I × ส่วนลด% (J)
N = แถม → I × แถม/(ซื้อ+แถม)
Q = VAT = I × 7% (ถ้าโหมด VAT)
R = (I − ส่วนลดเงินสด H − K − N) + Q + ค่าขนส่ง O
```
### 6.2 ราคาขายคำนวณ
```
V (ส่ง×หน่วยใหญ่) = ตัวคูณ E × R × (1 + ส่ง%)
X (ปลีกคำนวณ)     = R × (1 + ปลีก%)
```
### 6.3 ราคาโหล
```
ราคาโหล = R × (1 + ส่ง%) × 12
บาร์โค้ดโหล = "5" + บาร์โค้ดชิ้น
```

## 7. สัญญากับ Backend (ฟังก์ชันเดิมที่ UI ใหม่ต้องเรียก — reuse)
| ฟังก์ชัน | ตอนไหน | ส่ง/รับ |
|---|---|---|
| `getDropdownData()` | เปิดหน้า | รับ: หมวด/หน่วย/ผู้ขาย/นิติ |
| `searchProductsFromSheet(q)` | ค้นหา | รับ: รายการสินค้า (ทุกคอลัมน์ A–AI) |
| `processAndSaveAll(imgBase64, barcodeA, info)` | กด Save | ส่ง: รูป + บาร์โค้ด + ค่าทุกฟิลด์ (ดู payload ข้อ 8) |
| `manageListData(key, old, new, mode)` | ปุ่ม ⚙️ | เพิ่ม/แก้ dropdown |

> **ห้ามเปลี่ยน input/output ของ 4 ฟังก์ชันนี้** — ดีไซน์ใหม่แค่จัดหน้าตา แล้วต่อสายกลับฟังก์ชันเดิม

## 9. โจทย์เฉพาะมือถือ (ฝาก Designer)
- ฟอร์มยาว (3 บล็อก + ป๊อปต้นทุน) → ออกแบบให้ scroll ลื่น/แบ่ง step บนจอเล็ก
- ปุ่ม ▲▼ 0.25 และปุ่ม Save ต้องโดนง่ายด้วยนิ้วโป้ง (sticky ปุ่ม Save ด้านล่าง?)
- กล้องถ่ายรูป/สแกน เป็นปุ่มเด่น
- แสงจ้า: คอนทราสต์สูง ตัวเลขราคาใหญ่ชัด
- สถานะ "รออัปเดต/ครบ" ควรเห็นชัดในรายการ เพื่อรู้ว่าเหลือสำรวจกี่ตัว

## 10. สิ่งที่ mock ปัจจุบันยังขาด (ต้องเพิ่มในดีไซน์)
1. **หน้าฟอร์มแก้ไขทั้งหมด** (บล็อก A/B/C) — mock มีแค่หน้ารายการ
2. **ป๊อปอัปคำนวณต้นทุน** (กรอกราคาซื้อ/ส่วนลด/แถม/ขนส่ง/VAT → R)
3. **ป๊อปจัดการ dropdown** (⚙️ เพิ่มหมวด/หน่วย/นิติ)
4. **ตัวคูณ %ส่ง/%ปลีก** ที่อยู่ในป๊อปต้นทุน (mS/mT)
5. นิยาม KPI + เงื่อนไข "ครบ" ให้ชัด
