# 1. ปรัชญาโครงสร้างข้อมูล: Master + Logs

```
                 ┌──────────────────────────────┐
                 │  MASTER DATA  (= KLH DATA เดิม)│   ← สถานะปัจจุบัน
                 │  ราคา / รายละเอียด / ต้นทุนล่าสุด│      (อ่านเร็ว)
                 └──────────────▲───────────────┘
                                │ sync (สรุปจาก log)
   ┌───────────┬───────────┬───┴───────┬───────────┬───────────┐
   ▼           ▼           ▼           ▼           ▼           ▼
สินค้าเข้า   ราคาเปลี่ยน  ต้นทุนเปลี่ยน  การขาย     เจ้าหนี้     ลูกหนี้
STOCK_LOG  PRICE_LOG   COST_LOG    SALES_*     AP_LEDGER   AR_LEDGER
(IN)       +FIFO_BATCH                STOCK_LOG(OUT)
```

**กฎทอง:** Log = ความจริง (append-only ห้ามแก้), Master = ภาพสรุปล่าสุด (rebuild ได้จาก log ด้วย `recalcStockFromLog`)

| Log | มี/ต้องทำ | หมายเหตุ |
|---|---|---|
| สินค้าเข้า | ✅ STOCK_LOG (Type=IN) + FIFO_BATCH | มีแล้ว |
| การขาย | ✅ STOCK_LOG (Type=OUT) + SALES_HEADER/DETAIL | มีแล้ว |
| ราคาเปลี่ยน | 🟡 PRICE_HISTORY (มีบางส่วนจาก OCR) | ต้องทำ PRICE_LOG ให้ครบ → sync KLH DATA |
| ต้นทุนเปลี่ยน | 🔲 COST_LOG | ทำใหม่ → เก็บทุกครั้งต้นทุนขยับ |
| เจ้าหนี้ | ✅ AP_LEDGER | มีแล้ว |
| ลูกหนี้ | ✅ AR_LEDGER | มีแล้ว |

> หมายเหตุ: KLH DATA จะเปลี่ยนชื่อเป็น MASTER DATA ภายหลัง — ตอนนี้ใช้ชื่อเดิมไปก่อน (กันสับสน)

# 2. เครือข่ายคลัง (Warehouse Network)

ออกแบบให้แต่ละคลังมี **บทบาท + ข้อจำกัด + ต้นทุนการไปเอา** ต่างกัน → ระบบใช้ข้อมูลนี้ตัดสินใจวางของ

| รหัส | ชื่อคลัง | บทบาท | ความจุ | น้ำหนัก | ระยะ/ค่าไปเอา | เก็บอะไร |
|---|---|---|---|---|---|---|
| **W1** | คลังหน้าร้าน | Pick-face ขายหน้าร้าน | เล็ก | จำกัด | 0 (อยู่ร้าน) | ของขายทุกตัว เก็บ **3–5 วันพอขาย** |
| **W2** | ข้างบ้าน+บนบ้าน | Buffer ชั่วคราว / ขายส่งบ่อย | เล็กมาก | **เบาเท่านั้น** | ใกล้ | ของเบา หมุนเร็ว / buffer |
| **W3** | ซอยโทรศัพท์ | Fast-mover + **ของชื่อ KLH** | กลาง (≈2× เล็ก) | ปานกลาง | ใกล้-กลาง | ของขายเร็ว / สินค้าหจก.เคแอลเอช (ง่ายต่อบัญชีสรรพากร) |
| **W4** | โรงสี | Bulk reserve | **ใหญ่** | หนักได้ | **ไกล (ไปเอาแพง)** | ของชิ้นใหญ่/หนัก, สต๊อกสำรองก้อนใหญ่ |
| **W5** | (อนาคต) รวมกระจาย | Distribution Center | ใหญ่ | — | — | กระจายสู่สาขา เมื่อขยาย |

**คุณสมบัติคลังที่ต้องเก็บเพิ่มใน WAREHOUSE sheet:**
`WH_ID | WH_Name | Role | Capacity_Units | Weight_Limit_Kg | Fetch_Cost | Distance_Min | Allow_Heavy | Priority | Active`

- `Role`: FRONT / BUFFER / FAST / BULK / DC
- `Fetch_Cost`: ต้นทุนต่อรอบไปเอา (ใช้ถ่วงน้ำหนักการตัดสินใจ — โรงสีสูง)
- `Priority`: ลำดับที่ระบบเลือกหยิบมาเติมหน้าร้าน (W2/W3 ก่อน W4)

# 3. ตรรกะตัดสินใจ "วางของคลังไหน" (Slotting / Allocation)

ใช้คุณสมบัติสินค้า (จาก Master + ABC + ยอดขาย) จับคู่กับคุณสมบัติคลัง:

```
สำหรับสินค้าแต่ละตัว:
  velocity   = ยอดขายเฉลี่ย/วัน (จาก SALES log)
  abcClass   = A/B/C (Pareto)
  weight     = น้ำหนัก/หน่วย (ต้องเพิ่ม field ใน Master)
  entity     = ชื่อนิติ (KLH DATA col AD)
  size       = ชิ้นใหญ่/เล็ก

กฎจัดวาง (เรียงความสำคัญ):
  1. ของหนัก/ใหญ่           → W4 โรงสี (เก็บก้อนใหญ่) + เผื่อ W1 พอขาย
  2. ของชื่อ "KLH"          → W3 ซอยโทรศัพท์ (แยกชัด ทำบัญชีสรรพากรง่าย)
  3. Class A หมุนเร็ว        → W3 (สำรอง) + W1 (หน้าร้านเต็มเสมอ)
  4. ของเบา หมุนเร็ว         → W2 (buffer ใกล้ เติมไว)
  5. Class C ช้า/Dead        → ไม่เก็บหน้าร้านเยอะ, กองที่ W4, พิจารณาระบาย
  6. ทุกตัวที่ขาย            → W1 ต้องมี ≥ ROP เสมอ (3–5 วัน)
```

**ผลลัพธ์:** ระบบแนะนำ "ควรเก็บ SKU นี้ที่คลังไหน เท่าไร" + เตือนเมื่อวางผิดที่ (เช่น ของหนักไปอยู่ W2)

# 4. การเติมหน้าร้าน (Replenishment Flow)

```
ขายที่ POS → STOCK_LOG OUT (W1) → W1 ลดลง
   ↓
W1 < ROP (3–5 วัน)?
   ↓ ใช่
สร้าง PICK_LIST อัตโนมัติ: หยิบจากคลังสำรองที่ใกล้สุด/ถูกสุด
   ลำดับ: W2 → W3 → W4 (ตาม Priority + Fetch_Cost)
   ↓
พนักงานหยิบของ → completePickList → TRANSFER W2→W1
   ↓
W1 เต็มพอขายอีก 3–5 วัน
```

- ROP หน้าร้าน = ยอดขายเฉลี่ย/วัน × จำนวนวันเป้าหมาย (3–5) + safety stock
- รวม pick หลาย SKU เป็นรอบเดียว (ลดจำนวนเที่ยว = ประหยัด โดยเฉพาะ W4 ไกล)

# 5. หลักวิชาการ Inventory (ที่เลือกทำทั้งหมด)

### 5.1 ABC Analysis (Pareto 80/20)
- จัดอันดับ SKU ตาม **มูลค่าขายต่อปี** (velocity × ราคา)
- A = สะสม 80% แรก, B = 80–95%, C = 95–100%
- ใช้กำหนดความเข้มในการคุม: A เช็คถี่/safety สูง, C ปล่อยหลวม
- ข้อมูล: SALES log + ราคา (มีครบ)

### 5.2 EOQ — ปริมาณสั่งซื้อประหยัดสุด
```
EOQ = √(2 × D × S / H)
  D = ดีมานด์ต่อปี (จาก SALES log)
  S = ค่าใช้จ่ายต่อการสั่ง 1 ครั้ง (ค่าขนส่ง/เวลา — ตั้งใน CONFIG)
  H = ค่าถือครองต่อหน่วยต่อปี (= ต้นทุน × %ถือครอง เช่น 20%)
```
→ บอกว่าควรสั่งทีละกี่ชิ้นถึงคุ้ม (ไม่สั่งถี่เกิน/ไม่ตุนเกิน)

### 5.3 Safety Stock + ROP อัตโนมัติ
```
Safety Stock = Z × σ_demand × √Lead_Time
  Z = ระดับบริการ (95% → Z≈1.65)
  σ_demand = ส่วนเบี่ยงเบนยอดขายรายวัน (จาก SALES log)
  Lead_Time = เวลาสั่ง→รับ (วัด จาก PO/invoice log)

ROP = (ดีมานด์เฉลี่ย/วัน × Lead_Time) + Safety Stock
```
→ แทนการตั้ง ROP มือใน SKU_WH_CONFIG ด้วยค่าคำนวณจริง (อัปเดตทุกเดือน)

### 5.4 Dead Stock / Turnover
```
Inventory Turnover = ต้นทุนขาย/ปี ÷ มูลค่าสต๊อกเฉลี่ย
DOH (Days On Hand) = 365 ÷ Turnover
Dead stock = ไม่มี OUT ใน X วัน (เช่น 90 วัน)
```
→ รายงานของค้างทุน → ระบาย/ลดสั่ง คืนเงินสด

# 6. ช่องว่างที่ต้องผูก (Integration Gaps)

| # | ปัญหา | วิธีแก้ |
|---|---|---|
| 1 | POS `closeSale` เขียน stock ตรง ไม่ผ่าน WMS | ให้ closeSale เรียก logic เดียวกับ `consumeFifo_`+`updateBal_`+`checkAllRop` |
| 2 | OCR รับเข้าไม่เข้า WMS | ใบกำกับซื้อ → `batchReceiveFromInvoice` (มีฟังก์ชันแล้ว แค่ยังไม่เรียก) |
| 3 | Key SKU vs barcode ไม่ตรง | ตกลงใช้ `barcode` (ITEM-xxxx) เป็นกุญแจเดียวทั้งระบบ |
| 4 | ROP ตั้งมือ | แทนด้วย ROP คำนวณ (ข้อ 5.3) |
| 5 | WAREHOUSE ยังเป็นชื่อ generic (W1–W5 สาขา 2–5) | เปลี่ยนเป็นชื่อจริง + เพิ่ม field คุณสมบัติ (ข้อ 2) |
| 6 | ยังไม่มี field น้ำหนัก/ขนาด ใน Master | เพิ่ม Weight, Volume เพื่อใช้ slotting |
